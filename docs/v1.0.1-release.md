## Switch `runtimeVersion` policy from `appVersion` to `fingerprint`

**File:** `app.json` → `expo.runtimeVersion.policy`

**Why now:** with `appVersion` policy in v1.0, every time we bump
`expo.version` (e.g. 1.0.0 → 1.0.1) the previous binaries in the wild
become unreachable by EAS Update — they only accept OTAs whose
`runtimeVersion` matches their own `1.0.0`. We also have no protection
against accidentally publishing an OTA whose JS calls into a native
module the binary doesn't ship.

`fingerprint` policy fixes both: `runtimeVersion` becomes a hash of
everything that affects the native build (deps, plugins, app.json
native config). Pure-JS edits leave the hash unchanged so the OTA flows
across `version` bumps, and any native-affecting change shifts the hash
so the OTA correctly **doesn't** reach old binaries.

**Why we deferred to v1.0.1:** the policy switch only takes effect on
binaries built *after* the change. v1.0 build (9) was built with
`appVersion`, so it carries `runtimeVersion: "1.0.0"` forever. Switching
the policy doesn't strand it any worse than it already is — but
introducing the change inside the very first release window adds a
moving part Apple reviewers can poke at. Cleaner to land it at the top
of the v1.0.1 cycle, ship the first fingerprint-based native build
(`1.0.1` bumped in app.json), and confirm OTAs flow on that build
before relying on it.

**Steps when picking it up:**
1. Edit `app.json`:
   ```json
   "runtimeVersion": { "policy": "fingerprint" }
   ```
2. `npx expo-doctor` to confirm no related warnings.
3. Bump `expo.version` to `1.0.1` in `app.json` (the v1.0.1 release bump
   regardless).
4. `eas build --profile production --platform ios` — this is the build
   that anchors the new fingerprint runtime version.
5. After App Store approval, every JS-only fix from then on can ship
   via the manual OTA workflow without further native builds — until a
   native input changes.

## Re-enable Android + web in the OTA workflow

`.github/workflows/eas-update.yml` currently passes `--platform ios`
to `eas update` because v1.0 ships iOS-only and bundling for Android
+ web added ~1m to every dispatch with no consumer. When the Android
build / web export comes online, drop the flag (or change to
`--platform all`) so OTA fan-outs to every platform from one run.
